00017 // A few variations on a theme of the "star" graph representation by
00018 // Ebert, as described in J. Ebert, "A versatile data structure for
00019 // edge-oriented graph algorithms."  Communications of the ACM
00020 // 30(6):513-519 (June 1987).
00021 // http://portal.acm.org/citation.cfm?id=214769
00022 //
00023 // In this file there are three representations that have much in
00024 // common. The general one, called simply EbertGraph, contains both
00025 // forward- and backward-star representations. The other, called
00026 // ForwardEbertGraph, contains only the forward-star representation of
00027 // the graph, and is appropriate for applications where the reverse
00028 // arcs are not needed.
00029 //
00030 // The point of including all the representations in this one file is
00031 // to capitalize, where possible, on the commonalities among them, and
00032 // those commonalities are mostly factored out into base classes as
00033 // described below. Despite the commonalities, however, each of the
00034 // three representations presents a somewhat different interface
00035 // because of their different underlying semantics. A quintessential
00036 // example is that the AddArc() method, very natural for the
00037 // EbertGraph representation, cannot exist for an inherently static
00038 // representation like ForwardStaticGraph.
00039 //
00040 // Many clients are expected to use the interfaces to the graph
00041 // objects directly, but some clients are parameterized by graph type
00042 // and need a consistent interface for their underlying graph
00043 // objects. For such clients, a small library of class templates is
00044 // provided to give a consistent interface to clients where the
00045 // underlying graph interfaces differ. Examples are the
00046 // AnnotatedGraphBuildManager<> template, which provides a uniform
00047 // interface for building the various types of graphs; and the
00048 // TailArrayManager<> template, which provides a uniform interface for
00049 // applications that need to map from arc indices to arc tail nodes,
00050 // accounting for the fact that such a mapping has to be requested
00051 // explicitly from the ForwardStaticGraph and ForwardStarGraph
00052 // representations.
00053 //
00054 // There are two base class templates, StarGraphBase, and
00055 // EbertGraphBase; their purpose is to hold methods and data
00056 // structures that are in common among their descendants. Only classes
00057 // that are leaves in the following hierarchy tree are eligible for
00058 // free-standing instantiation and use by clients. The parentheses
00059 // around StarGraphBase and EbertGraphBase indicate that they should
00060 // not normally be instantiated by clients:
00061 //
00062 //                     (StarGraphBase)                       |
00063 //                       /         \                         |
00064 //                      /           \                        |
00065 //                     /             \                       |
00066 //                    /               \                      |
00067 //           (EbertGraphBase)     ForwardStaticGraph         |
00068 //            /            \                                 |
00069 //           /              \                                |
00070 //      EbertGraph     ForwardEbertGraph                     |
00071 //
00072 // In the general EbertGraph case, the graph is represented with three
00073 // arrays.
00074 // Let n be the number of nodes and m be the number of arcs.
00075 // Let i be an integer in [0..m-1], denoting the index of an arc.
00076 //  * head_[i] contains the end-node of arc i,
00077 //  * head_[-i-1] contains the start-node of arc i.
00078 // Note that in two's-complement arithmetic, -i-1 = ~i.
00079 // Consequently:
00080 //  * head_[~i] contains the start-node of the arc reverse to arc i,
00081 //  * head_[i] contains the end-node of the arc reverse to arc i.
00082 //  Note that if arc (u, v) is defined, then the data structure also stores
00083 //  (v, u).
00084 //  Arc ~i thus denotes the arc reverse to arc i.
00085 //  This is what makes this representation useful for undirected graphs and for
00086 //  implementing algorithms like bidirectional shortest paths.
00087 //  Also note that the representation handles multi-graphs. If several arcs
00088 //  going from node u to node v are added to the graph, they will be handled as
00089 //  separate arcs.
00090 //
00091 // Now, for an integer u in [0..n-1] denoting the index of a node:
00092 //  * first_incident_arc_[u] denotes the first arc in the adjacency list of u.
00093 //  * going from an arc i, the adjacency list can be traversed using
00094 //    j = next_adjacent_arc_[i].
00095 //
00096 // The EbertGraph implementation has the following benefits:
00097 //  * It is able to handle both directed or undirected graphs.
00098 //  * Being based on indices, it is easily serializable. Only the contents
00099 //    of the head_ array need to be stored. Even so, serialization is
00100 //    currently not implemented.
00101 //  * The node indices and arc indices can be stored in 32 bits, while
00102 //    still allowing to go a bit further than the 4-gigabyte
00103 //    limitation (48 gigabytes for a pure graph, without capacities or
00104 //    costs.)
00105 //  * The representation can be recomputed if edges have been loaded from
00106 //  * The representation can be recomputed if edges have been loaded from
00107 //    external memory or if edges have been re-ordered.
00108 //  * The memory consumption is: 2 * m * sizeof(NodeIndexType)
00109 //                             + 2 * m * sizeof(ArcIndexType)
00110 //                             + n * sizeof(ArcIndexType)
00111 //    plus a small constant.
00112 //
00113 // The EbertGraph implementation differs from the implementation described in
00114 // [Ebert 1987] in the following respects:
00115 //  * arcs are represented using an (i, ~i) approach, whereas Ebert used
00116 //    (i, -i). Indices for direct arcs thus start at 0, in a fashion that is
00117 //    compatible with the index numbering in C and C++. Note that we also tested
00118 //    a (2*i, 2*i+1) storage pattern, which did not show any speed benefit, and
00119 //    made the use of the API much more difficult.
00120 //  * because of this, the 'nil' values for nodes and arcs are not 0, as Ebert
00121 //    first described. The value for the 'nil' node is set to -1, while the
00122 //    value for the 'nil' arc is set to the smallest integer representable with
00123 //    ArcIndexSize bytes.
00124 //  * it is possible to add arcs to the graph, with AddArc, in a much simpler
00125 //    way than described by Ebert.
00126 //  * TODO(user) although it is already possible, using the
00127 //    GroupForwardArcsByFunctor method, to group all the outgoing (resp.
00128 //    incoming) arcs of a node, the iterator logic could still be improved to
00129 //    allow traversing the outgoing (resp. incoming) arcs in O(out_degree(node))
00130 //    (resp. O(in_degree(node))) instead of O(degree(node)).
00131 //  * TODO(user) it is possible to implement arc deletion and garbage collection
00132 //    in an efficient (relatively) manner. For the time being we haven't seen an
00133 //    application for this.
00134 //
00135 // The ForwardEbertGraph representation is like the EbertGraph case described
00136 // above, with the following modifications:
00137 //  * The part of the head_[] array with negative indices is absent. In its
00138 //    place is a pointer tail_ which, if assigned, points to an array of tail
00139 //    nodes indexed by (nonnegative) arc index. In typical usage tail_ is NULL
00140 //    and the memory for the tail nodes need not be allocated.
00141 //  * The array of arc tails can be allocated as needed and populated from the
00142 //    adjacency lists of the graph.
00143 //  * Representing only the forward star of each node implies that the graph
00144 //    cannot be serialized directly nor rebuilt from scratch from just the head_
00145 //    array. Rebuilding from scratch requires constructing the array of arc
00146 //    tails from the adjacency lists first, and serialization can be done either
00147 //    by first constructing the array of arc tails from the adjacency lists, or
00148 //    by serializing directly from the adjacency lists.
00149 //  * The memory consumption is: m * sizeof(NodeIndexType)
00150 //                             + m * sizeof(ArcIndexType)
00151 //                             + n * sizeof(ArcIndexType)
00152 //    plus a small constant when the array of arc tails is absent. Allocating
00153 //    the arc tail array adds another m * sizeof(NodeIndexType).
00154 //
00155 // The ForwardStaticGraph representation is restricted yet farther
00156 // than ForwardEbertGraph, with the benefit that it provides higher
00157 // performance to those applications that can use it.
00158 //  * As with ForwardEbertGraph, the presence of the array of arc
00159 //    tails is optional.
00160 //  * The outgoing adjacency list for each node is stored in a
00161 //    contiguous segment of the head_[] array, obviating the
00162 //    next_adjacent_arc_ structure entirely and ensuring good locality
00163 //    of reference for applications that iterate over outgoing
00164 //    adjacency lists.
00165 //  * The memory consumption is: m * sizeof(NodeIndexType)
00166 //                             + n * sizeof(ArcIndexType)
00167 //    plus a small constant when the array of arc tails is absent. Allocating
00168 //    the arc tail array adds another m * sizeof(NodeIndexType).

00642     // a pointer, because we modify its referent.

00647     // used for two purposes related to degree before setting up its

00686       }

00704       } else {

00706         // the head_ array in a single loop.
00707         for (ArcIndexType input_arc = 0; input_arc < num_arcs; ++input_arc) {

00709           NodeIndexType head = input_arcs[input_arc].second;
00710           ArcIndexType arc = first_incident_arc_[kFirstNode + tail];

00712           head_[kFirstArc + arc] = kFirstNode + head;

00714       }

00718     // there is a special sentry element at the end of
00719     // first_incident_arc_.

00721          node > /* kFirstNode */ 0;

00746           ArcIndexType end_offset = (end > 0 ? 1 : 0);

00749           begin = end;
00750         }
00751       }
00752     }

01750   // method is called. Therefore we have this method also, and we
01751   // ensure that it is called only when tail_ is guaranteed to have
01752   // been initialized.
01753   void ReserveTailArray(ArcIndexType new_max_num_arcs) {
01754     if (tail_ != NULL) {
01755       // The (*tail_) values are already canonical, so we're just
01756       // reserving additional space for new arcs that haven't been
01757       // added yet.
01758       if (tail_->Reserve(kFirstArc, new_max_num_arcs - 1)) {
01759         for (ArcIndexType arc = tail_->max_index() + 1;
01760              arc < new_max_num_arcs;
01761              ++arc) {
01762           tail_->Set(arc, kNilNode);
01763         }
01764       }
01765     }
01766   }
01767 
01768   // Reserves space for the arrays indexed by arc indices, except
01769   // (*tail_) even if it is present. We cannot grow the (*tail_) array
01770   // in this method because this method is called from
01771   // Base::Reserve(), which in turn is called from the base template
01772   // class constructor. That base class constructor is called on *this
01773   // before tail_ is constructed. Hence when this method is called,
01774   // tail_ might contain garbage. This method can safely refer only to
01775   // fields of the base template class, not to fields of *this outside
01776   // the base template class.
01777   //
01778   // The strange situation in which this method of a derived class can
01779   // refer only to members of the base class arises because different
01780   // derived classes use the data members of the base class in
01781   // slightly different ways. The purpose of this derived class
01782   // method, then, is only to encode the derived-class-specific
01783   // conventions for how the derived class uses the data members of
01784   // the base class.
01785   //
01786   // To be specific, the forward-star graph representation, lacking
01787   // reverse arcs, allocates only the positive index range for the
01788   // head_ and next_adjacent_arc_ arrays, while the general
01789   // representation allocates space for both positive- and
01790   // negative-indexed arcs (i.e., both forward and reverse arcs).
01791   void ReserveInternal(NodeIndexType new_max_num_nodes,
01792                        ArcIndexType new_max_num_arcs) {
01793     head_.Reserve(kFirstArc, new_max_num_arcs - 1);
01794     next_adjacent_arc_.Reserve(kFirstArc, new_max_num_arcs - 1);
01795     for (ArcIndexType arc = max_num_arcs_; arc < new_max_num_arcs; ++arc) {
01796       head_.Set(arc, kNilNode);
01797       next_adjacent_arc_.Set(arc, kNilArc);
01798     }
01799     ReserveTailArray(new_max_num_arcs);
01800   }

01821     DCHECK(CheckArcValidity(arc));
01822     DCHECK(IsNodeValid(tail));

02106 // Builds a directed line graph for 'graph' (see "directed line graph" in
02107 // http://en.wikipedia.org/wiki/Line_graph). Arcs of the original graph
02108 // become nodes and the new graph contains only nodes created from arcs in the
02109 // original graph (we use the notation (a->b) for these new nodes); the index
02110 // of the node (a->b) in the new graph is exactly the same as the index of the
02111 // arc a->b in the original graph.
02112 // An arc from node (a->b) to node (c->d) in the new graph is added if and only
02113 // if b == c in the original graph.
02114 // This method expects that 'line_graph' is an empty graph (it has no nodes
02115 // and no arcs).
02116 // Returns false on an error.
02117 template<typename GraphType> bool BuildLineGraph(const GraphType& graph,
02118                                                  GraphType* const line_graph) {
02119   if (line_graph == NULL) {
02120     LOG(DFATAL) << "line_graph must not be NULL";
02121     return false;
02122   }
02123   if (line_graph->num_nodes() != 0) {
02124     LOG(DFATAL) << "line_graph must be empty";
02125     return false;
02126   }
02127   typedef typename GraphType::ArcIterator ArcIterator;
02128   typedef typename GraphType::OutgoingArcIterator OutgoingArcIterator;
02129   // Sizing then filling.
02130   typename GraphType::ArcIndex num_arcs = 0;
02131   for (ArcIterator arc_iterator(graph);
02132        arc_iterator.Ok();
02133        arc_iterator.Next()) {
02134     const typename GraphType::ArcIndex arc = arc_iterator.Index();
02135     const typename GraphType::NodeIndex head = graph.Head(arc);
02136     for (OutgoingArcIterator iterator(graph, head);
02137          iterator.Ok();
02138          iterator.Next()) {
02139       ++num_arcs;
02140     }
02141   }
02142   line_graph->Reserve(graph.num_arcs(), num_arcs);
02143   for (ArcIterator arc_iterator(graph);
02144        arc_iterator.Ok();
02145        arc_iterator.Next()) {
02146     const typename GraphType::ArcIndex arc = arc_iterator.Index();
02147     const typename GraphType::NodeIndex head = graph.Head(arc);
02148     for (OutgoingArcIterator iterator(graph, head);
02149          iterator.Ok();
02150          iterator.Next()) {
02151       line_graph->AddArc(arc, iterator.Index());
02152     }
02153   }
02154   return true;
02155 }
